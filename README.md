# FundTrace — Financial Fraud & Mule Account Tracking System

A prototype investigative tool for visualizing fund flow across bank accounts,
detecting mule-account patterns (fan-in, fan-out, rapid pass-through,
structuring), and managing linked case files. Built as a working demo for
pitching to a cyber crime / financial intelligence unit.

---

## ⚠️ Read this before you pitch this anywhere

This is a **working prototype on synthetic data**, not a system connected to
real bank, NPCI, RBI, or law-enforcement databases. That distinction matters
a lot when you present it:

- **What's real:** the full application — UI, risk-scoring logic, fund-flow
  graph engine, fan-in/fan-out detection, case management — all genuinely
  functional code you can run, demo, and extend.
- **What's simulated:** the data. All 17 accounts, 20 transactions, and 2
  case files are fictional, generated by `backend/scripts/seed.js`. No real
  person, account number, or bank is represented.
- **The honest pitch:** *"This is a working prototype of the analysis engine
  and interface. Given access to [specific data source — bank SARs, NPCI
  transaction logs, etc.] under proper legal authorization, this is how it
  would plug in."* Departments take that framing seriously. Claiming or
  implying it's already live with real financial data is the fastest way to
  get the whole project dismissed the moment someone asks a follow-up
  question — and could create real legal exposure depending on what you
  claim.

See **"Path to production"** at the bottom for what real integration would
actually require.

## What it does

- **Dashboard** — system-wide stats: accounts monitored, confirmed mule
  accounts, flagged transaction volume, risk distribution, flag-type
  breakdown.
- **Account Registry** — searchable/filterable list of every monitored
  account with computed risk score and status (Active / Under Review /
  Confirmed Mule / Frozen).
- **Account Detail** — full transaction history (sent + received), risk
  flags with severity and description, linked case files, investigator
  notes.
- **Transaction Ledger** — every recorded transaction, filterable to
  flagged-only.
- **Fund Flow Map** — interactive force-directed graph. Node size/color
  encodes risk level; red edges mark flagged transactions. Click any node to
  jump to its account detail, or trace outward from a specific account to
  see only its connected subnetwork (this is the feature that actually maps
  to "show me where this money went").
- **Case Files** — investigations with linked accounts tagged by role
  (Victim / Intermediary / Mule).

### Detection logic baked into the seed data

The synthetic dataset is built to demonstrate the patterns real investigators
look for:

| Pattern | What it looks like | Where to see it |
|---|---|---|
| **Rapid pass-through** | Funds forwarded within minutes of receipt | Vikram Singh Rathore's account |
| **Fan-out** | One large inflow split into several outgoing transfers | Sunita Devi Patel's account |
| **Fan-in** | Several unrelated people send money into one account | Deepak Chauhan's account |
| **Structuring** | Amounts kept just under round/reportable thresholds | Same fan-out transfers above |
| **New-account high-volume** | Account opened days ago, suddenly moves large sums | Most mule accounts in the dataset |
| **Identity red flags** | Deceased holder, fabricated KYC docs | Arvind Kumar Yadav, Mohammed Iqbal Ansari |

Risk scores are computed by summing severity-weighted flags per account
(see `computeRisk()` in `backend/scripts/seed.js`) — this logic is meant as
a clear, explainable starting point you can extend with real rules.

## Tech stack

- **Backend:** Node.js + Express
- **Data layer:** flat JSON file (`backend/db/data.json`), generated by a
  seed script — no native database dependency, so there's nothing to
  compile and it runs identically on any machine with Node installed. See
  "Path to production" for swapping this for a real database.
- **Frontend:** vanilla HTML/CSS/JS, no framework, no build step. The fund
  flow graph is a custom lightweight force-directed renderer (no charting
  library) so the risk-coloring and flagged-edge behavior are exact.

## How to run it on your system

### 1. Install Node.js

You need Node.js 18 or later. Check if you already have it:

```bash
node --version
```

If not installed, download from [nodejs.org](https://nodejs.org) (LTS
version) and install it — Windows/Mac/Linux all have installers.

### 2. Get the project files onto your machine

If you received this as a folder, just place `fund-tracker/` anywhere on
your system. The structure should look like:

```
fund-tracker/
├── backend/
│   ├── db/
│   ├── routes/
│   ├── scripts/
│   ├── server.js
│   └── package.json
└── frontend/
    ├── css/
    ├── js/
    └── index.html
```

### 3. Install backend dependencies

Open a terminal in the `backend/` folder:

```bash
cd fund-tracker/backend
npm install
```

This installs Express and CORS — both pure JavaScript, no compilation step.

### 4. Generate the sample data

```bash
npm run seed
```

You should see:
```
✅ Seed complete.
   Accounts: 17
   Transactions: 20
   Flags: 12
   Cases: 2
```

This creates `backend/db/data.json`. Re-run this command any time you want
to reset the data back to its original state.

### 5. Start the server

```bash
npm start
```

You should see:
```
  FundTrace backend running → http://localhost:4000
```

### 6. Open it in your browser

Go to **http://localhost:4000**

That's it — the frontend is served by the same Express server, so there's
no separate frontend build or server to run.

### To stop the server

Press `Ctrl + C` in the terminal where it's running.

```
backend/
  server.js              Express app entry point, mounts API routes + static frontend

  css/style.css              All styling
  js/
    api.js                   Fetch wrapper for backend calls
```

## API reference

| Method | Endpoint | Purpose |
|---|---|---|
| GET | `/api/stats` | Dashboard summary numbers |
| GET | `/api/accounts?search=&status=&risk_level=` | Filtered account list |
| GET | `/api/accounts/:id` | Full account detail incl. flags, transactions, cases |
| GET | `/api/transactions?flagged_only=&min_amount=&channel=` | Filtered transaction list |
| GET | `/api/transactions/graph` | Full network graph (all nodes/edges) |
| GET | `/api/transactions/graph/:accountId?depth=2` | Subgraph traced outward from one account |
| GET | `/api/cases` | All case files |
| GET | `/api/cases/:id` | Case detail with linked accounts and roles |

## Extending this for a real pitch

Things worth adding before a serious presentation, roughly in order of
impact:

1. **CSV import** — let the audience upload a real (anonymized/sample) bank
   statement and watch the graph populate live. This is usually more
   convincing than a pre-loaded demo.
2. **Configurable risk rules** — expose the flag thresholds (e.g. "flag if
   >80% of balance withdrawn within 24h") as settings instead of hardcoded
   weights, so an analyst can tune it.
3. **Export to PDF/case report** — investigators need to hand off findings;
   a "generate case summary" button goes a long way.
4. **Authentication & audit logging** — anything handling financial
   investigation data needs login, role-based access, and a log of who
   viewed what, before it's near real accounts.
5. **Path to production data integration** — the real conversation to have
   with a department is about RBI/NPCI's existing fraud-reporting channels
   (like the **National Cybercrime Reporting Portal** and bank-side
   **Central Payments Fraud Information Registry**) and how a tool like this
   would consume structured exports from them under a formal MoU — not
   scraping or directly connecting to bank systems, which an individual or
   external team cannot legally do.

## Limitations to state up front

- Single-user, no authentication — fine for a demo, not for real account data.
- JSON file storage handles this dataset fine; a real deployment needs a
  proper database (PostgreSQL is the natural next step) and would benefit
  from cross-bank record matching that a flat file can't do at scale.
- Risk scoring is a simple additive model for demonstration; production
  fraud scoring should be validated against historical confirmed-fraud
  cases, ideally with input from the bank's existing fraud/AML team.
